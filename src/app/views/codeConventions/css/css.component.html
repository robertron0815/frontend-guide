CSS / SCSS Coding Guideline

Sass or SCSS was developed to maintain an overview in larger projects. In addition come various features and extras that extend CSS. There is also a risk of writing confusing code if you don't follow certain rules.

1. Valid and readable code

The fundamental guidelines for css is written here:

  • Google Coding Guidlines
  • 4 spaces block indentation

2. SCSS best practice

It's not recommended to use scss-files inside of angular components. On the one hand, there is always a risk to write redundant code. On the other hand you need to include special setting or mixin files for every component file.
If you use the way that is shown below, you need to import the files only one time and there are no duplicate code.

  • there are one style-file for the whole project, whichs includes all component- and setting-files:
    • Angular: style.scss
    • React: app.scss
  • create one scss file per module/component (if neccessary)
    • create a structure like components/mixins...
    • naming file: "_compontent.scss"
    • import in style.scss as "@import 'assets/styles/components/header';"

3. Units and values

Best practice for font-sizes is to set the value by the rem-unit. The rem-value is relative to the html or body font-size (default 16px) and could be scaled very easliy. For better reading you can use the rem-calc-mixin, which you can find in this repository (asstes/styles/mixins/_rem-calc.scss).

Example:
font-size: rem-calc(14px); or just
font-size: rem-calc(14);

For container you can use absolute values like px or relative values like %, vh, vw, rem...

4. Naming

A good naming convention will tell you and your team:

  • what type of thing a class does
  • where a class can be used
  • what (else) a class might be related to

Class names are divided by hyphens "-". Camelcase, PascalCase and underscores* are not allowed to use.
* Except the usage in BEM.

4.1 BEM - Block Element Modifier

In large projects it is recommended to use the BEM-naming

  • Block: The sole root of the component
  • Element: A component part of the Block
  • Modifier: A variant or extension of the Block

5. Mobile first

Mobile first means that we scale our application design from small to large. This concept provides that the version optimized for mobile devices is created first and successive extensions take place.

6. Linting

For better control about the sass/scss code, it's necessary to use a linter. From now on it's it's standard for every new project to integrate the sass-lint package.

There are only a few steps for implementing:

  • include sass-lint.yml in your root-folder
  • install sass-lint and husky via npm or yarn. Sass-lint is the linter. Husky is a pre-commit hook library. It prevents commiting your changes if there are errors or warnings.
    • npm install -g sass-lint
    • npm install husky
  • add commands to your package.json (have a look to image 1 and 2)

Add sass-lint command more readable to your npm script section.

It is not possible to commit your code to git, if there are warnings or errors.

  • It is useful to ignore third-party libraries from linting.
  • Now you can run the script via npm run sass-lint and watch your console. Good luck ;-)
